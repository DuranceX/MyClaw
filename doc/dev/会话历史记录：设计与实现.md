# 会话历史记录：设计与实现

这篇文档记录会话历史持久化功能的完整实现，参考了 Claude Code 的 JSONL 文件存储方案。

---

## 背景与动机

原始实现中，消息完全存在内存里，刷新页面后全部丢失。用户需要：

- 刷新页面后自动恢复上次对话
- 左侧边栏展示历史会话列表，支持切换
- 多会话并行，随时新建或删除

---

## 参考：Claude Code 的设计

Claude Code 用 JSONL 格式持久化会话，核心思路：

- 每个会话对应一个 `<sessionId>.jsonl` 文件，每行一条消息
- 单独维护一个 `sessions.json` 索引文件，记录所有会话的元数据（id、标题、时间）
- 消息结构包含 `uuid`、`parentUuid`、`session_id`、`timestamp` 等字段，支持链式重建

本项目做了简化：不需要链式 parentUuid，消息按顺序存储即可；标题从第一条用户消息自动提取，无需 LLM 生成。

## 存储结构

```
PROJECT_ROOT/
└── .chat-sessions/
    ├── sessions.json          # 索引文件
    ├── sess-abc123.jsonl      # 会话消息文件
    └── sess-def456.jsonl
```

**sessions.json 格式**（按 `updated_at` 降序排列）：

```json
[
  {
    "id": "sess-abc123",
    "title": "帮我写一个 Python 脚本…",
    "created_at": "2025-01-01T10:00:00+00:00",
    "updated_at": "2025-01-01T10:05:00+00:00"
  }
]
```

**JSONL 消息文件格式**（每行一条 AI SDK UIMessage）：

```jsonl
{"role": "user", "id": "msg-xxx", "parts": [{"type": "text", "text": "你好"}]}
{"role": "assistant", "id": "msg-yyy", "parts": [{"type": "step-start"}, {"type": "text", "text": "你好！"}]}
{"role": "error", "id": "err-1234567890", "parts": [{"type": "text", "text": "LLM request failed: 400 ..."}]}
```

消息格式直接使用 AI SDK 的 UIMessage 格式，这样 `setMessages(loadedMessages)` 可以无缝恢复前端状态，不需要任何转换。

错误消息用 `role: "error"` 标记，是对标准格式的扩展，前端渲染时单独处理。

Props：

### chat.tsx 核心改动

**session_id 生成**：

```typescript
function newSessionId() {
  return `sess-${crypto.randomUUID().replace(/-/g, '').slice(0, 12)}`;
}
```

页面初始化时生成一个新 id，切换/新建会话时重新生成。

**传递 session_id 给后端**：

```typescript
const { messages, sendMessage, setMessages, status, error } = useChat({
  body: { session_id: currentSessionId },
});
```

**保存时机**：监听 `status` 变化，从 `streaming/submitted` 变为 `ready` 或 `error` 时触发保存：

```typescript
const prevStatus = useRef(status);
useEffect(() => {
  const wasLoading = prevStatus.current === 'streaming' || prevStatus.current === 'submitted';
  const isNowDone = status === 'ready' || status === 'error';
  prevStatus.current = status;
  if (!wasLoading || !isNowDone || messages.length === 0) return;

  // 出错时追加一条 role: 'error' 消息，保留错误现场
  const msgsToSave = status === 'error' && error
    ? [...messages, { id: `err-${Date.now()}`, role: 'error', parts: [{ type: 'text', text: error.message }] }]
    : messages;

  fetch(`/api/sessions/${currentSessionId}`, {
    method: 'PUT',
    body: JSON.stringify({ messages: msgsToSave }),
  }).then(() => setSidebarRefresh(n => n + 1));
}, [status]);
```

**自动恢复上次会话**：页面加载时拉取列表，自动选中第一条（`updated_at` 最新）：

```typescript
useEffect(() => {
  fetch('/api/sessions')
    .then(r => r.json())
    .then(sessions => {
      if (sessions.length > 0) loadSession(sessions[0].id);
    });
}, []);
```

**错误消息渲染**：历史记录中 `role: 'error'` 的消息渲染为红色错误卡片，与实时错误提示样式一致。

## 数据流

```
用户发送消息
    │
    ▼
useChat 发起 POST /api/chat（body 含 session_id）
    │
    ▼
流式响应结束（status → ready / error）
    │
    ▼
前端 PUT /api/sessions/{id}（携带完整 messages）
    │
    ▼
后端写入 .chat-sessions/{id}.jsonl
更新 sessions.json 索引
    │
    ▼
sidebarRefresh++ → SessionSidebar 重新拉取列表
```

---

## 关键设计决策

**为什么由前端主动保存，而不是后端流结束时保存？**

后端的 `stream_chat` 只拿到请求时的 `payload.messages`（不含本轮 AI 回复）。AI 的回复是通过 SSE 流逐步发给前端的，后端没有完整的最终消息列表。让前端在流结束后主动 PUT 是最简单的方案，不需要后端重建 AI 消息。

**为什么用 JSONL 而不是 JSON 数组？**

JSONL 支持追加写入（append-only），每行独立，单行损坏不影响其他行。Claude Code 也是同样的选择。本项目目前是覆盖写入（每次保存全量），但格式上保持了 JSONL 的兼容性，未来可以改为追加模式。

**为什么错误用 `role: 'error'` 而不是单独字段？**

保持消息列表结构统一，加载历史时不需要额外处理，直接 `setMessages` 即可，错误消息会按顺序出现在正确位置。
