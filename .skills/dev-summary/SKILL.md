# Dev Summary（开发过程总结文档生成）

## 概述

本技能用于将一次开发过程（对话讨论 + 代码改动）整理成一份结构化的中文开发文档，保存到项目的 `docs/dev/` 目录中，方便日后回溯设计思路和实现细节。

---

## 工作流程

### 第一步：确认主题

从用户的指令中提取文档主题（即"xxx"部分）。如果主题不明确，询问用户：

> "你想总结哪个功能或改动？请简单描述一下主题。"

### 第二步：收集素材

从以下来源收集信息，**不要遗漏任何一项**：

**2.1 对话历史**

回顾当前对话中的所有内容，重点提取：
- 用户提出的问题和需求
- 讨论过的多种方案及其优劣对比
- 最终选择了哪个方案，以及选择的理由
- 过程中遇到的问题、报错、困惑
- 解决问题的思路和方法

**2.2 代码改动**

运行以下命令获取本次改动的上下文：

```bash
# 查看未提交的改动
git diff HEAD

# 查看最近几次提交（根据对话时间范围判断取几条）
git log --oneline -10

# 查看具体提交的改动
git show <commit_hash>
```

如果用户提到了特定文件，直接读取这些文件了解当前实现。

### 第三步：分析并组织内容

将收集到的素材整理成以下几个维度：

1. **背景与动机**：为什么要做这个改动？解决了什么问题？
2. **设计思路**：核心设计理念是什么？
3. **方案对比**（如有）：讨论过哪些方案？各自的优缺点？最终选择了哪个？
4. **具体改动**：实际修改了哪些文件？关键代码逻辑是什么？
5. **遇到的问题与解决方案**：过程中踩了哪些坑？怎么解决的？

如果某个维度在对话和代码中都没有体现，可以省略该章节，不要强行填充。

### 第四步：生成文档

按照下面的模板生成 Markdown 文档，**语言使用中文**：

```markdown
# {文档标题}

{一句话概括这次改动的核心内容}

当前相关代码位于：

- `{文件路径}` — {简要说明}
- ...

---

## 背景与动机

{描述为什么要做这个改动，解决了什么问题或满足了什么需求}

## 设计思路

{描述核心设计理念，可以用类比或对比来说明}

## 方案对比

> 如果没有讨论过多个方案，删除本章节。

### 方案 A：{方案名}

{描述方案，说明优缺点}

### 方案 B：{方案名}

{描述方案，说明优缺点}

**最终选择**：{选择了哪个方案，以及原因}

## 具体改动

### {模块或文件名}

{说明这个文件/模块做了什么改动，关键逻辑是什么}

```{语言}
{关键代码片段（如有必要）}
```

## 遇到的问题与解决方案

> 如果过程顺利没有遇到问题，删除本章节。

### 问题：{问题描述}

{描述遇到了什么问题，错误信息是什么}

**解决方案**：{如何解决的}
```

### 第五步：确定文件名并写入

文件名规则：
- 使用中文，简洁描述主题
- 格式：`{主题}.md`
- 例如：`工具系统重构：buildTool模式.md`、`会话历史记录：设计与实现.md`

写入路径：`docs/dev/{文件名}.md`

写入后告知用户：

> "文档已写入 `docs/dev/{文件名}.md`"

---

## 注意事项

- **忠实于对话**：文档内容应来自实际的对话和代码，不要凭空编造细节。
- **保留设计意图**：不只是记录"做了什么"，更要记录"为什么这么做"。
- **代码片段要精简**：只引用能说明问题的关键片段，不要大段粘贴代码。
- **省略空章节**：如果某个章节没有对应内容，直接删除，不要留空或写"无"。
- **文档风格参考**：可以读取 `docs/dev/` 下已有的文档，保持风格一致。
