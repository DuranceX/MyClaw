# Requirement Review（需求总结报告生成）

## 概述

本技能用于将一个需求从"文档 + 代码"两个维度进行交叉分析，最终输出一份结构化的中文总结报告。

核心输入有三项（第 2 项可选）：

| # | 输入 | 来源 | 是否必须 |
|---|------|------|----------|
| 1 | **需求文档** | TAPD 链接 或 用户直接粘贴的 PRD 文本 | ✅ 必须 |
| 2 | **技术方案** | 腾讯文档链接 或 用户直接粘贴的方案文本 | ❌ 可选 |
| 3 | **代码分支名** | 用户告知的 Git 分支名称 | ✅ 必须 |

最终输出：一份包含 **需求概述 → 技术方案摘要 → 代码变更分析 → 风险点/建议 → 总结** 五个章节的中文 Markdown 报告。

---

## 工作流程

### 第一步：收集输入

主动向用户确认以下信息（如果对话中尚未提供）：

1. **需求文档**：请提供 TAPD 链接，或直接粘贴 PRD 内容。
2. **技术方案**（可选）：如果有技术方案文档，请提供腾讯文档链接或直接粘贴。
3. **代码分支**：请告诉我本次需求对应的 Git 分支名称。

> 不要一次性抛出所有问题。如果用户已经在对话中提供了部分信息，只追问缺失的部分。

### 第二步：解析需求文档

根据输入类型分别处理：

**情况 A — TAPD 链接**：
- 使用 `fetch_external_resources`（kind: `tapd`）抓取 TAPD 内容。
- 如果抓取失败或内容不完整，请求用户手动粘贴关键内容。

**情况 B — 用户粘贴的 PRD 文本**：
- 直接在对话中解析文本内容。

**解析目标**（提取以下关键信息）：
- 需求标题 / 名称
- 需求背景与目标
- 功能范围（做什么、不做什么）
- 核心业务流程 / 用户故事
- 验收标准（如有）
- 关联方 / 上下游依赖（如有）

### 第三步：解析技术方案（可选）

如果用户提供了技术方案：

**情况 A — 腾讯文档链接**：
- 使用 `fetch_external_resources`（kind: `info`）抓取文档内容。
- 如果抓取失败，请求用户手动粘贴关键内容。

**情况 B — 用户粘贴的方案文本**：
- 直接在对话中解析。

**解析目标**：
- 整体架构设计思路
- 核心模块 / 接口设计
- 数据模型变更（如有）
- 依赖的外部服务 / 基础设施变更
- 兼容性 / 迁移方案（如有）

### 第四步：分析代码变更

1. **确认基准分支**：默认与 `main` 对比，如果 `main` 不存在则尝试 `master`。
2. **获取变更概览**：
   ```bash
   git fetch origin <branch_name>
   git diff origin/main...origin/<branch_name> --stat
   ```
3. **获取详细 diff**（按文件逐个分析，优先关注核心变更）：
   ```bash
   git diff origin/main...origin/<branch_name> -- <file_path>
   ```
4. **获取提交记录**：
   ```bash
   git log origin/main..origin/<branch_name> --oneline --no-merges
   ```

**分析维度**：
- **变更范围**：涉及哪些模块 / 目录 / 文件？新增、修改、删除各多少？
- **核心逻辑变更**：主要改了什么业务逻辑？新增了哪些接口/组件/函数？
- **数据层变更**：是否有数据库 schema 变更、新增字段、迁移脚本？
- **配置变更**：环境变量、配置文件、依赖包是否有变化？
- **测试覆盖**：是否有新增 / 修改测试用例？

### 第五步：交叉分析 & 生成报告

将需求文档、技术方案与实际代码变更进行交叉对比：

- **需求覆盖度**：需求中描述的功能点，在代码中是否都有对应实现？
- **方案一致性**：代码实现是否与技术方案的设计一致？有无偏差？
- **额外变更**：代码中是否有需求/方案未提及的变更？（可能是重构、bugfix、技术债等）
- **遗漏风险**：是否有需求提到但代码中未体现的功能？

## 边界情况处理

### TAPD / 文档抓取失败
- 先重试一次。
- 仍失败则告知用户抓取失败，请求手动粘贴需求/方案的核心内容。
- 不要在没有需求内容的情况下强行生成报告。

### 分支不存在或无差异
- 如果指定分支在远端不存在，提示用户确认分支名是否正确。
- 如果 diff 为空（分支与 main 无差异），告知用户并确认是否要继续。

### 技术方案未提供
- 这是可选输入，未提供时在报告的"技术方案摘要"章节标注"用户未提供技术方案文档，已跳过"。
- 不要因为缺少技术方案而阻塞报告生成。

### 代码变更量过大
- 如果 diff 超过 500 个文件或总变更量极大，优先分析核心业务代码变更。
- 忽略自动生成的文件（如 lock 文件、编译产物、vendor 目录）。
- 在报告中注明已过滤的内容。

### 需求与代码不匹配
- 如果发现代码变更明显与需求文档不对应，在报告中明确指出，不要强行关联。
- 在"风险点"中标注，并建议用户确认。

## 注意事项

1. **报告语言**：始终使用中文撰写报告。
2. **客观中立**：报告应基于事实分析，避免主观臆断。对不确定的地方用"可能"、"建议确认"等措辞。
3. **隐私意识**：报告中不要暴露敏感信息（如密钥、token、内部 IP 等），如在代码中发现应在风险点中提示。
4. **渐进式输出**：如果内容较多，可以分步输出（先概览再详解），而不是让用户等待很长时间。
5. **可追溯性**：核心结论尽量附上对应的文件路径和行号，方便用户回溯验证。
